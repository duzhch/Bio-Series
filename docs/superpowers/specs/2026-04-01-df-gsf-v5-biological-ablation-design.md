# DF_GSF_v5 Biological Module Ablation Design

## Goal

Add a module-level ablation mechanism to `DF_GSF_v5` so we can quantify how much predictive gain comes from biologically meaningful components of the full `bio_master_v11` pipeline.

The baseline is the current full model. The feature is not intended to compare historical model versions. It is intended to compare biologically interpretable variants of the same model family under the same train/test split and result collection workflow.

## Scope

This design covers:

- a unified `--ablation` runtime argument on `DF_GSF_v5.py run-all`
- ablation-aware result directory naming
- six fixed ablation modes centered on biological priors and population structure
- minimal changes to the training pipeline so existing workflow remains usable
- batch submission support for ablation runs
- a dedicated ablation summary script

This design does not include:

- model-internal architectural ablations beyond the agreed biological scope
- changes to historical result directories
- broad refactoring of the current training code

## Baseline and Ablation Modes

The baseline is `bio_master_v11` with all currently active components enabled:

- wide tower
- deep tower
- delta embeddings
- gene2vec embeddings
- PCA/context tower
- orthogonality term between deep and context features

The `--ablation` argument will accept the following fixed values:

- `full`
- `no_delta`
- `no_gene2vec`
- `no_bio_prior`
- `no_pca`
- `pca_only_prior_off`

`full` is the default and must preserve current behavior as closely as possible.

## Precise Semantics

### `full`

Current `bio_master_v11` behavior.

Deep tower uses genotype plus biological prior channel derived from delta and gene2vec. Context tower uses PCA features. Orthogonality loss remains enabled.

### `no_delta`

Delta embeddings are generated by the pipeline for compatibility, but the model must not use delta information.

Expected model behavior:

- prior generator uses gene2vec only
- delta contribution is replaced by a zero tensor before fusion
- context tower remains enabled

Interpretation:

- contribution of sequence semantic prior learned from PigBERT delta embeddings

### `no_gene2vec`

Gene2Vec features are generated by the pipeline for compatibility, but the model must not use gene2vec information.

Expected model behavior:

- prior generator uses delta only
- gene2vec contribution is replaced by a zero tensor before fusion
- context tower remains enabled

Interpretation:

- contribution of gene functional or co-expression prior

### `no_bio_prior`

The deep tower must run without any biological prior channel.

Expected model behavior:

- deep tower becomes genotype-only for its local convolution input
- input channel count changes from two channels to one channel
- context tower remains enabled
- orthogonality loss remains enabled because context tower still exists

Interpretation:

- contribution of biological prior information as a whole compared with a pure SNP deep model

### `no_pca`

Disable the PCA/context tower inside the prediction model.

Expected model behavior:

- context tower is bypassed
- PCA features loaded for training are replaced by zeros or omitted consistently
- context output is removed from the prediction sum
- orthogonality loss is disabled because context features are absent

Important boundary:

- the upstream GWAS step may still keep its current PCA-based correction
- this ablation is defined as removing explicit population-structure modeling from the prediction model, not redesigning the GWAS preprocessing stage

Interpretation:

- contribution of explicit population structure correction during prediction

### `pca_only_prior_off`

Keep model structure close to the full model, but neutralize the biological prior content.

Expected model behavior:

- context tower remains enabled
- deep tower still receives two input channels
- genotype channel remains unchanged
- prior channel is replaced with an all-zero neutral channel
- prior generator output is not used as biological information

Interpretation:

- isolates the contribution of prior content while holding the two-channel deep-tower input structure constant

This mode exists specifically to separate two effects:

- information content from biological priors
- architectural effect of having an auxiliary prior channel

## Recommended Interpretation Matrix

The intended comparisons are:

- `full` vs `no_delta`: contribution of sequence-context prior
- `full` vs `no_gene2vec`: contribution of gene-function prior
- `full` vs `no_bio_prior`: contribution of biological priors overall
- `full` vs `no_pca`: contribution of population-structure correction in the prediction model
- `full` vs `pca_only_prior_off`: contribution of biological prior content when deep input structure is held constant
- `pca_only_prior_off` vs `no_bio_prior`: contribution of the extra prior channel structure itself

## Command-Line Interface

Extend `DF_GSF_v5.py run-all` with:

```bash
python DF_GSF_v5.py run-all \
  --config config/global_config.yaml \
  --dataset LargeWhite_Pop1 \
  --trait BF \
  --rep rep_01 \
  --model bio_master_v11 \
  --ablation no_delta
```

Rules:

- `--ablation` defaults to `full`
- invalid values should fail fast with a clear error
- the argument should also be accepted by helper steps such as `run-gwas` for path consistency even if the ablation does not affect GWAS behavior directly

## Result Directory Layout

To avoid overwriting existing runs, ablation results must use dedicated folder names.

Recommended layout:

- `results/<dataset>_<trait>/bio_master_v11__full/rep_01`
- `results/<dataset>_<trait>/bio_master_v11__no_delta/rep_01`
- `results/<dataset>_<trait>/bio_master_v11__no_gene2vec/rep_01`
- `results/<dataset>_<trait>/bio_master_v11__no_bio_prior/rep_01`
- `results/<dataset>_<trait>/bio_master_v11__no_pca/rep_01`
- `results/<dataset>_<trait>/bio_master_v11__pca_only_prior_off/rep_01`

This naming scheme preserves:

- compatibility with current `dataset_trait/model/rep` discovery patterns
- easy grouping by model plus ablation
- safety for already completed `bio_master_v11` runs

## Run Metadata

Each ablation result directory should contain `run_meta.json` with at least:

- `dataset`
- `trait`
- `rep`
- `model`
- `ablation`
- `result_dir`
- `delta_path`
- `gene_path`
- `plink_prefix`
- `context_enabled`
- `bio_prior_mode`
- `orthogonality_enabled`

This file is for traceability and easier downstream summarization.

## Implementation Plan Boundary

Only four code areas need to change.

### 1. `DF_GSF_v5.py`

Add `--ablation` to shared CLI arguments.

Update `infer_paths()` so the result folder name includes both model and ablation. For example:

- `bio_master_v11__full`
- `bio_master_v11__no_delta`

Pass `ablation` into the selected training function.

Write `run_meta.json` before or after training completes.

### 2. `src/models/bio_master_v11.py`

This is the main implementation site.

Required changes:

- define valid ablation names and normalize config
- make `PriorGenerator` usage ablation-aware
- support three deep prior modes:
  - normal learned prior from delta plus gene2vec
  - zeroed prior content with two-channel structure retained
  - pure genotype-only single-channel deep input
- make context tower optional
- disable orthogonality loss when context tower is disabled
- keep output API stable enough for existing training flow

The training entry point should accept `ablation='full'`.

### 3. `submit_jobs.py`

Current script generation should be expanded to support:

- `--model`
- `--ablations`

The generated command must include both `--model` and `--ablation`.

Important correction:

- current generated commands do not explicitly pass `--model`, which can silently fall back to the parser default
- this should be fixed as part of the ablation work

### 4. `scripts/compare_ablations.py`

Add a new comparison script dedicated to ablations rather than reusing the existing cross-model comparison script.

The new script should:

- discover `bio_master_v11__*` directories
- use `full` as anchor
- collect per-rep PCC and MSE
- compute per-ablation delta PCC relative to `full`
- write rep-level and summary-level CSV outputs

## Data Flow and Caching Strategy

For stability and minimal disruption, the pipeline should continue to generate delta and gene2vec files for all ablation modes except where a future optimization explicitly skips them.

Reasons:

- avoids changing upstream feature engineering control flow during the first implementation
- preserves compatibility with current file expectations
- reduces the chance of introducing path-dependent bugs

This means the first version of the feature focuses on usage ablation, not feature-generation ablation.

## Error Handling

The implementation should fail fast for:

- unknown ablation names
- incompatible tensor dimensions after ablation logic
- missing metadata needed for summary collection

The implementation should not silently map one ablation name to another.

## Verification Strategy

Verification should happen at two levels.

### Lightweight shape and forward verification

Use small synthetic tensors to confirm that all six ablation modes:

- instantiate successfully
- run a forward pass
- return compatible output shapes
- do not produce immediate runtime errors when context or prior channels are disabled

### Small end-to-end smoke test

Run a shortened training job on a single existing dataset/trait/rep with:

- `full`
- `no_bio_prior`
- `no_pca`

Confirm that each mode produces:

- separate result directory
- `stats.json`
- `pred.csv`
- `run_meta.json`

Also confirm that the new ablation comparison script can summarize those runs.

## Backward Compatibility

The design aims to preserve existing behavior when:

- `--ablation` is omitted
- `--ablation full` is used

Historical result directories should remain untouched.

## Risks

### Interpretation risk

If `no_bio_prior` and `pca_only_prior_off` are conflated, the experiment cannot distinguish between prior information content and prior-channel architecture. The implementation must preserve the difference.

### Maintenance risk

If each ablation is implemented as a separate model file, behavior will drift. All ablations should stay under one model implementation with shared training logic.

### Workflow risk

If batch scripts do not include `--model` and `--ablation` explicitly, submitted jobs may run the wrong configuration while appearing successful.

## Out of Scope for First Pass

- skipping delta or gene2vec feature extraction to save runtime
- changing GWAS preprocessing semantics for `no_pca`
- adding statistical significance tests to ablation summaries
- adding visualization dashboards
